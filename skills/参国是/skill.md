# 参国是 — 政策信息追踪与解读技能

## 目标

围绕用户指定的政策议题，通过政策领域专属搜索策略与多源交叉比对，产出三种形式的输出：**政策汇编**（全面汇总）、**政策简报**（精炼摘要）、**政策解析**（宏观视角下的深度解读）。

**本技能的独特价值**：在通用信息汇编（揽风云）和深度意识形态批判（见真章）之间，提供一个**制度化、历史性的中观分析层**——既不是纯罗列，也不做价值批判，而是用长周期和宏观结构的视角为政策提供新的理解框架。

**本技能不进行事实真伪判断**——那是 破虚妄（`/破虚妄`）的职责。本技能不进行以权力批判为核心的意识形态分析——那是 见真章（`/见真章`）的职责。

---

## 运行模式

| 模式 | 用法 | 输出结构 | 适用场景 |
|------|------|----------|----------|
| `compile`（默认） | `/参国是` 或 `/参国是 --mode compile` | 政策汇编报告（6 章节） | 政策汇总、多领域政策梳理、政策时间线 |
| `brief` | `/参国是 --mode brief` | 政策简报（3 章节） | 快速了解政策要点、政务信息摘要 |
| `insight` | `/参国是 --mode insight` | 政策解析报告（5 章节） | 深度理解某一政策的背景、逻辑与影响 |

**混合模式**：解析模式不一定单独使用。用户可在前两种模式中加入 `--with-insight` 参数，使报告在汇编/简报基础上，对关键政策附加宏观视角解析。

| 混合用法 | 效果 |
|----------|------|
| `/参国是 --mode compile --with-insight` | 汇编报告中，对重点政策条目附加"解读卡片" |
| `/参国是 --mode brief --with-insight` | 简报末尾附加一段"关键信号解读" |

若用户指令中无法判断模式，默认使用 `compile`。若用户需求是"解读""评论""怎么看""意味着什么"，自动选择 `insight`。

## 参数

| 参数 | 效果 |
|------|------|
| `--mode compile` | 政策汇编报告（默认） |
| `--mode brief` | 政策简报 |
| `--mode insight` | 政策解析报告 |
| `--with-insight` | 在 compile 或 brief 模式中混入解析视角 |
| `--scope` | 限定范围，如 `--scope 国家级`、`--scope 浙江省`、`--scope 2026Q1` |
| `--level` | 限定政策层级，如 `--level 法律`、`--level 行政法规`、`--level 地方性法规` |
| `--timeline` | 在 compile 模式下输出为政策时间线（按日期排列） |

## 政策领域知识

本技能内置以下领域知识，在搜索和信息组织阶段自动应用。

### 一、政策文件效力层级

| 层级 | 名称 | 发布主体 | 文号特征 | 识别优先级 |
|------|------|----------|----------|------------|
| L1 | 法律 | 全国人大及其常委会 | 主席令第X号 | 最高 |
| L2 | 行政法规 | 国务院 | 国令第X号 / 国务院令 | 高 |
| L3 | 地方性法规 | 省级/设区市人大 | X省/市人大常委公告 | 高 |
| L4 | 部门规章 | 国务院各部委 | 部令第X号 | 中 |
| L5 | 地方政府规章 | 省级/设区市政府 | X省政府令 | 中 |
| L6 | 规范性文件 | 各级行政机关 | 国发/国办发/部发/省发等 | 中低 |
| L7 | 标准/指南 | 标准化机构/部委 | GB/GB/T、YD/等 | 参考 |

> 在汇编和简报中，应对每项政策标注效力层级。解析模式下，效力层级是判断政策力度的重要维度——同一领域内，提升效力层级（如从规范性文件升格为行政法规）本身就是重要信号。

### 二、关键发布节点与搜索窗口

| 时间节点 | 政策类型 | 搜索策略 |
|----------|----------|----------|
| **3月（两会）** | 立法草案、政府工作报告、年度计划 | 搜索"十四届全国人大X次会议 + 政策领域" |
| **国务院常务会议（每周）** | 行政法规、改革方案、试点批复 | 搜索"国务院常务会议 + 日期 + 研究/部署" |
| **月底/季末** | 规范性文件集中发布 | 以月份为窗口搜索部门网站 |
| **中央深改委会议** | 重大改革顶层设计 | 搜索"中央深改委第X次会议" |
| **中央政治局会议** | 季度经济定调、重大战略 | 搜索"中央政治局会议 + 月份" |
| **1号文件窗口** | 农业/农村领域年度文件 | 搜索"中央一号文件/省委一号文件 + 年份" |

### 三、政策搜索策略

**通用搜索词模式**：

```
国家级：site:gov.cn "政策关键词" 2026
部委级：site:miit.gov.cn | site:ndrc.gov.cn | site:mof.gov.cn ... "政策关键词"
省级：site:zj.gov.cn "政策关键词" 2026
政策全文："{政策名称}" filetype:pdf OR "全文"
配套细则："{国家政策名}" "贯彻落实" OR "实施意见" OR "配套措施"
```

**优先信源（按可信度和覆盖率排序）**：
1. 中国政府网政策库 (gov.cn/zhengce) — 国务院及国办文件最全
2. 各部委官网政策文件栏目 — 部门规章和规范性文件
3. 全国人大网 (npc.gov.cn) — 法律全文和立法说明
4. 省级政府网站政策文件栏目 — 地方性法规和政府规章
5. 新华社/央视网 — 政策发布的首发报道
6. 司法部法律法规数据库 — 跨部门检索

**搜索两轮制**（与揽风云共享原则）：
- 首轮：2-4 次并行搜索，覆盖不同发布主体、不同政策领域
- 补充：针对缺口定向搜索，1-2 次
- 核心信息需 2 个以上独立来源交叉确认

**外媒信源（条件触发）**：

外媒搜索默认不激活。仅当模型判断用户需求涉及**国际事务/境外视角**时启用（如：国际条约对中国的影响、外国对华政策、境外对中国政策的反应、国际比较分析）。

触发后，外媒信源获取遵循 揽风云「外媒信源清单与获取策略」的完整工作流——WebSearch snippet（主力，A/B 级）+ GitHub 搜索/聚合仓库（C 级恢复）+ ≥2 源三角确认。不走 WebFetch（全外媒 L1 阻断）。政策分析场景的适用外媒：

| 媒体 | 政策分析强项 | 可用性 |
|------|-------------|--------|
| **Reuters** | 全球经济政策、贸易制裁动态 | A 级（转载路径） |
| **AP News** | 美国国内政策过程、国会动态 | A 级 |
| **BBC News** | 国际关系、外交政策全球视角 | A 级 |
| **The Guardian** | 人权、环境政策、社会政策 | A 级 |
| **CNN** | 美国政治、白宫政策动态 | A 级 |
| **Politico** | 美国立法过程、监管政策细节 | A 级 |
| **SCMP** | 中国政策外部影响、亚太地缘 | A 级 |
| **Foreign Policy** | 外交政策分析、国际安全 | A 级 |
| **France 24 / AFP** | 欧洲政策视角、非洲政策 | A 级（转载路径） |
| **NYT** | 美国政策深度调查 | B 级（snippet 薄） |
| **Bloomberg** | 产业政策、能源政策、金融监管 | B 级（转载片段） |
| **WSJ** | 商业监管、贸易政策 | B 级（间接） |
| **FT / Economist / Foreign Affairs / Nikkei Asia / The Atlantic** | — | C 级（GitHub 可部分恢复，详见揽风云） |

外媒在政策报告中的定位为**补充视角**（国际反应、比较分析），非主要信源。外媒与国内官方信源冲突时，优先采信国内官方信源（政策分析的本土立场）。

### 四、央地政策联动追踪

当用户请求涉及某一领域时，自动追踪从中央到地方的实施链条：

```
国家层面法律/行政法规
  → 国务院/部委配套细则（实施意见、工作方案）
    → 省级地方性法规/政府规章（地方化落地）
      → 省直部门配套文件（操作细则）
```

汇编和简报中，若发现完整的央地联动链条，应单独标注为"政策落地链"。

## 工作流程

### 阶段 1：解析用户需求

从用户指令中提取：
- **政策议题/领域**：要追踪/解读的政策主题
- **地域范围**：全国 / 特定省份 / 跨区域
- **时间范围**：特定窗口 or 默认（近3个月）
- **政策层级**：是否限定法律/行政法规/部门规章等
- **模式判断**：compile（汇总）/ brief（简报）/ insight（解析）/ 混合

### 阶段 2：首轮搜索

并行执行 2-4 次 WebSearch，搜索词设计遵循：
- 至少一次 `site:gov.cn` 精确搜索
- 至少一次覆盖国务院/中办国办层面
- 若涉及地方政策，至少一次覆盖对应省级政府网站
- 搜索词之间有明确区分度（不同发布主体、不同关键词角度）

### 阶段 3：缺口识别与补充搜索

对比首轮结果，识别缺口并定向补充。停止条件同 揽风云：核心信息已被 2 个以上独立来源交叉确认，无明显的时间/地域/层级空白。

### 阶段 4：信息组织与标注

将收集到的政策信息按主题/领域组织。引用编号体系、来源分级、确定性标记制度**继承 揽风云 的完整体系**（编号 `[1]` 递增、A/B/C 三级来源、▲/●/△ 确定性标记）。

**政策条目的标准信息字段**：
- 政策名称（完整规范名称）
- 发布日期 / 施行日期
- 发布机构（联合发布的全部列出）
- 文号（可获取时）
- 效力层级（L1-L7）
- 关联政策（上位法/配套细则/替代的旧政策）

### 阶段 5：撰写报告

按对应模式的输出规范组织内容。

**若启用 `--with-insight`**，对于 compile/brief 中的重点政策条目，附加简短的"解读卡片"（2-4 句话），从宏观视角说明该政策的信号意义。解读卡片在正文中以缩进引用块或侧边栏样式呈现，与纯事实陈述明确区分。

## 解析模式方法论：六大宏观视角

解析模式（`--mode insight`）使用以下六大视角分析政策。**不需全部使用**，根据政策特点选择最富洞察力的 1-3 个。

### 视角一：历史演进

> 这项政策在同一个政策领域的长周期演变中处于什么位置？哪些是延续，哪些是真正的新变量？

- 追溯该领域的政策演变脉络（至少回看 5-10 年，重要的可拉到改革开放以来）
- 识别"变"与"不变"：措辞的微调、覆盖范围的扩大/缩小、力度的加减、执行主体变更
- 警惕"重新包装"——很多"新政"其实是旧政策的重新挂牌，指出这一点本身就是洞察

### 视角二：制度逻辑

> 这项政策为什么以这个形式、在这个时间、由这些部门联合发布？背后的制度性动因是什么？

- 发布机构的组合意味着什么（如：中办国办联合印发 = 最高政治意志；多部委联合 = 跨部门协调产物）
- 政策窗口的利用逻辑（如：为什么赶在两会前/后发布？为什么在某个事件后紧急出台？）
- 政策工具选择逻辑（为什么用法律而不是规范性文件？为什么用补贴而不是管制？）

### 视角三：国际比较

> 其他主要经济体/可比较国家在处理类似问题时采取了什么路径？中国选择的路径有何特征和由来？

- 寻找可比较的政策领域（不是找"谁更好"，而是找"不同选择背后的不同约束"）
- 指出中国路径的独特性及其制度/历史根源
- 避免简单化的"中国应该学X国"——重点在理解差异，不在评判优劣

### 视角四：利益结构

> 这项政策会对哪些群体产生什么影响？直接和间接的利益分配效应是什么？

- 绘制利益相关者图谱：直接受益者、间接受益者、可能受损者
- 分析政策的分配效应——不是做意识形态批判，而是中性地揭示"谁得到什么"
- 关注政策可能产生的二阶效应和意外后果

### 视角五：政策位序

> 这项政策在更大的政策组合中扮演什么角色？是收官之作、开路信号、补丁还是转向？

- 判断政策的"位序角色"：顶层设计 / 实施细则 / 试点方案 / 整改措施 / 应急响应
- 识别政策之间的逻辑关系：替代、补充、升级、对冲
- 关注该政策发布前后是否有"呼应的政策动作"，构成政策组合拳

### 视角六：周期节律

> 中国政策在特定领域是否存在可辨识的周期性？当前处于周期的哪个阶段？

- 识别该领域的政策周期特征（收紧→放松、中央集权→地方放权、扩张→整顿）
- 判断当前信号指向周期的哪个方向
- 注意：周期判断是有风险的推测性分析，需用"可能""信号显示"等措辞，不可包装为确定预测

### 解析模式的"度"

- **深度**：足够让读者获得"原来如此"的认知增益，但不陷入学术论文式的理论堆砌
- **判断力**：敢于提出明确的判断（"这一政策的真正变量不在X，而在Y"），但不过度断言
- **克制**：不对政策制定者的动机做恶意揣测；不进行意识形态标签化（"这是威权主义的……"之类）；不滑入见真章式的权力批判
- **可用性**：读者读完应能回答"这项政策为什么重要""它跟之前有什么不同""接下来应该关注什么"

## 输出规范

### 模式一：政策汇编报告（`--mode compile`，默认）

1. **任务概述** — 一句话概括用户需求和信息范围
2. **政策总览** — 按政策领域（经济/科技/环保/民生/法治/对外开放……）组织章节。每项政策条目含：名称、发布机构、日期、文号（可获取时）、效力层级标签 `[L1-L7]`。每条事实性陈述末尾以 `[编号]` 标注信源引用。**若用户指定 `--timeline`**，则按时间顺序排列，每项标注日期
3. **央地联动追踪**（若适用） — 列出从国家政策到地方配套的完整落地链路
4. **来源索引** — 编号列表。每条含：`[编号]` 来源名称、URL、等级（A/B/C）、类型。继承揽风云的完整索引格式
5. **信息覆盖完整性声明** — 已覆盖 vs 未覆盖的政策领域/层级/地域维度
6. **汇编时间戳** — 信息截止时间、汇编完成时间

**若启用 `--with-insight`**：在第 2 章的每个政策领域末尾，或被判定为重点政策的条目下方，附加"解读卡片"（以缩进引用块区分），从宏观视角提供 2-4 句解读。

### 模式二：政策简报（`--mode brief`）

1. **本期要目** — 3-5 句话概括本期最重要的政策动态
2. **政策清单** — 按领域分类的政策条目列表（仅含名称+日期+一句话要点+效力层级），精简至 compile 模式的约 30%-40% 篇幅
3. **来源列表** — 精简版索引（仅编号+名称+URL）

**若启用 `--with-insight`**：在第 1 章后附加一段"关键信号解读"（200-400 字），提炼本期政策中最重要的 1-2 个跨领域信号。

### 模式三：政策解析报告（`--mode insight`）

1. **政策背景** — 目标政策的基本信息（名称、发布机构、日期、效力层级、上位法/替代关系）
2. **核心内容概述** — 用 5-8 条要点概括政策的主要内容（不展开解读，只陈述）
3. **宏观视角解读** — 选择 1-3 个最富洞察力的视角展开分析。每个视角独立成小节，以视角名称为小标题（如"历史演进"、"制度逻辑"）。每节 200-400 字
4. **关键信号与后续关注** — 提炼该政策传递的最重要信号，列出后续应关注的政策动向（如：配套细则何时出台、试点效果评估时间窗口、是否会有地方跟进）
5. **来源索引** — 完整索引（同上）

### 混合模式的呈现约定

- **compile + insight**：解读卡片使用缩进引用块（`>` 样式或内缩边框），以"🔍 解读"为标签开头，明确区分于纯事实陈述
- **brief + insight**：关键信号解读以分隔线与正文分开，标注"本期信号"

## 与其他技能的关系

| 技能 | 做什么 | 与本技能的关系 |
|------|--------|----------------|
| **揽风云** | 通用信息汇编（任意主题） | 参国是 继承其搜索两轮制、引用编号体系、来源分级和确定性标记制度。揽风云 可以做政策汇编，但缺少政策层级识别、央地联动追踪和发布节点知识。对于非政策类汇编任务，使用 揽风云 |

## 导出

### 导出 HTML

**基础模板继承**：参国是的 HTML 输出继承 揽风云 `report_template.html` 的全部 CSS 类与结构约定。在此基础上叠加政策报告专属样式。

**文件输出**：写入用户指定目录；未指定则写入 `E:\AI\揽风云\`。文件名从标题自动生成（去非法字符，截断 40 字符），UTF-8 编码。

#### 必须遵守的 HTML 格式约束

以下规则适用于所有模式的 HTML 输出。违反任一条即为格式缺陷。

**1. 可点击引用链接**

正文中每条事实性陈述末尾的 `[编号]` 必须渲染为可点击链接，指向来源索引表中的对应行：

```html
<a href="#ref1" class="ref">[1]</a>
```

多源交叉确认标注多个编号时，每个编号独立链接：
```html
<a href="#ref1" class="ref">[1]</a><a href="#ref3" class="ref">[3]</a>
```

**2. 来源索引表格式**

来源索引必须使用 `<table>` 格式，每行以 `id="refN"` 作为锚点目标，并以 `class="source-A/B/C"` 标注来源等级：

```html
<table>
  <tr id="ref1" class="source-A">
    <td>[1]</td>
    <td>来源名称</td>
    <td>等级</td>
    <td><a href="...">链接</a></td>
  </tr>
</table>
```

来源等级 CSS 类（继承揽风云）：
- `.source-A { background:#e8f5e9; }` — 官方一手（绿底）
- `.source-B { background:#fff8e1; }` — 权威媒体（黄底）
- `.source-C { background:#fce4ec; }` — 商业/社交平台（粉底）

**3. 确定性标记**

具体数字/数据后必须内联确定性标记：
- `▲` — 官方渠道确认
- `●` — 多家独立媒体一致，未经官方文件确认
- `△` — 单一来源，或不同来源数字有出入

**4. 图例（Legend）**

来源索引末尾必须附加图例，解释标记符号含义：

```html
<div class="legend">
  <p><strong>确定性标记：</strong>▲ 官方确认 · ● 多源一致 · △ 单一来源或数字有出入</p>
  <p><strong>来源等级：</strong>A 一手官方 · B 权威媒体 · C 商业/社交平台</p>
  <p><strong>效力层级：</strong>L1 法律 → L2 行政法规 → L3 地方性法规 → L4 部门规章 → L5 地方政府规章 → L6 规范性文件 → L7 标准/指南</p>
</div>
```

**5. 图像支持**

为图像嵌入预留 CSS 类和 fallback 机制（继承揽风云）：

```html
<!-- 成功嵌入 -->
<div class="img-card" id="img1">
  <img src="data:image/jpeg;base64,..." alt="描述">
  <div class="caption">[图1] 图像描述 <a href="#refN" class="ref">[N]</a></div>
  <div class="meta">来源：出处名称 · 视觉验证状态：未验证 △</div>
</div>

<!-- 嵌入失败 -->
<div class="img-fallback">[图N] 嵌入失败，请访问<a href="原文URL">原文</a>查看</div>
```

**6. 元数据框**

使用 `<div class="meta-box"><dl>...</dl></div>` 格式展示结构化元数据：

```html
<div class="meta-box">
  <dl>
    <dt>信息截止时间</dt><dd>2026-06-16 12:00（北京时间）</dd>
    <dt>汇编完成时间</dt><dd>2026-06-16</dd>
    <dt>搜索轮次</dt><dd>2 轮</dd>
    <dt>引用信源数</dt><dd>20 条</dd>
  </dl>
</div>
```

**7. 章节分隔**

章节间使用 `<hr>` 分隔：

```html
</section>
<hr>
<section>
```

#### 参国是专属 CSS 增量

在继承揽风云基础 CSS 之上，政策报告需叠加以下样式（内联于 `<style>` 标签中，放在揽风云变量定义之后）：

| CSS 类/元素 | 用途 | 说明 |
|-------------|------|------|
| `blockquote` 增强 | 解读卡片 | 在揽风云基础 `blockquote` 上叠加暖色左边框（`#c9b99a`）和米色背景（`#f3f0ea`），以"🔍 解读"为标签开头 |
| `.provision-grid` | 条款卡片网格 | `display:grid; grid-template-columns:1fr 1fr;` 响应式两列布局，用于展示备忘录条款、政策要点等 |
| `.prov-card` | 单张条款卡片 | 带边框圆角的卡片容器，含 `.num`（编号）、`.title`（标题）、`.desc`（描述）三个子元素 |
| `.cmp-table` | 立场对照表 | 三列表格（议题 / 甲方立场 / 乙方立场），首列加粗 |
| `.challenge-card` | 挑战/风险卡片 | 带边框圆角的卡片，含 `.ch-title`（标题）+ `.ch-body`（正文） |

| `.summary-row` | 关键数字概览行 | `grid` 三列等高卡片，每张含 `.big`（大号数字）+ `.label`（说明标签） |
| `.timeline-table` | 时间线表格 | 带阶段分隔行（`.phase-row`，深色背景居中粗体）的完整表格 |
| `.tag-official / .tag-media / .tag-analysis` | 标签徽章 | 小号圆角色块，分别对应官方信息/媒体报道/分析评论 |
| `.meta-pill` | 元数据胶囊 | 圆角边框小标签，用于报告头部的模式/地域/时间等元信息 |
| `[L1]–[L7]` 标签 | 效力层级标签 | 内联样式：小号、等宽字体、对应层级颜色（L1 深红 → L7 浅灰），形如 `<span style="font-size:0.8em; font-family:monospace; color:#8b1a1a;">[L1]</span>` |

#### 解读卡片的 HTML 约定

compile/brief + insight 混合模式下，解读卡片使用 `<blockquote>` 并以 `<strong>🔍 解读：</strong>` 开头：


解读卡片必须紧跟在被解读的政策条目或章节之后，与纯事实陈述形成视觉区分（缩进 + 暖色左边框）。

## 注意事项

- **不做真伪判断**：不区分谣言与事实。若用户需要，引导至 `/破虚妄`。
- **不做意识形态批判**：不以权力批判为核心分析框架。若用户需要，引导至 `/见真章`。
- **解析克制**：解析模式追求制度性、历史性的洞察，避免将分析包装成预测，避免对政策制定者动机做恶意揣测。
- **效力层级可见**：对每项政策标注效力层级（L1-L7），让读者一眼判断政策的强制力和权威性。
- **央地链路完整**：涉及具体领域的政策汇编时，尽可能追踪从中央到地方的完整实施链。
- **信源偏好官方一手**：政策文件的搜索应以政府网站（.gov.cn）为首选信源，媒体报道为补充。
- **信息截止点**：必须在报告中明确标注信息截止时间，并注明搜索引擎索引延迟。
- **搜索节制**：控制在两轮以内。政策的"足够覆盖"比"穷尽"更重要。
- **HTML 格式合规**：导出 HTML 时必须严格遵守「导出 HTML」章节的全部格式约束——可点击引用链接、表格格式来源索引（含 `id="refN"` 锚点 + `source-A/B/C` 行级着色）、确定性标记内联、图例、图像支持预留。禁止以纯文本 `[N]` 代替链接、禁止以 `<div>` 代替 `<table>` 渲染来源索引、禁止遗漏图例。
- **外媒信源条件触发**：外媒搜索默认不激活。仅在用户需求涉及国际事务/境外视角时启用。启用后遵循 揽风云 的外媒获取工作流（WebSearch snippet 主力 + GitHub 补充 C 级 + 三角确认，不走 WebFetch）。C 级媒体通过 `gh search code` 和聚合仓库恢复，GitHub 单次调用 ≤3 次。
- **与 揽风云 的边界**：参国是 专注于政策和政务领域。当任务明显超出政策范畴（如企业新闻、娱乐事件、科技产品评测），应意识到这更适合 揽风云 处理。

## 扩展工具（规划中）

以下工具待使用频率确认后开发，当前仅做设计预留。开发触发条件：对应场景月均出现 ≥ 2 次。

### policy-registry — 跨会话政策注册表

**解决问题**：每次新会话从零开始搜索，长期追踪同一政策领域时反复搜到已覆盖的条目，浪费搜索轮次和上下文。

**机制**：每次汇编完成后，将已覆盖的政策条目（名称、日期、文号、发布机构、效力层级）追加写入本地 JSON 文件。新会话启动时读取，跳过已收录条目，聚焦补充新增政策。

**存储路径**：
```
~/.claude/skills/参国是/registry/
├── policies.json       # [{ 名称, 日期, 文号, 发布机构, 层级, 领域, 收录时间 }]
└── coverage_log.json   # [{ 月份, 领域, 地域, 收录条目数 }]
```

**交互约定**：Claude 不应假设 registry 是完整的。先搜索，再与 registry 比对去重，而非仅依赖 registry 判断覆盖范围。

### docnumber-parser — 文号批量解析器

**解决问题**：搜索结果批量返回政策文本时，手工逐条提取文号、日期、发布机构是纯体力劳动。

**功能**：从文本中提取标准政府公文文号并结构化输出。

**输入**：原始文本（搜索结果摘要或政策全文片段）

**输出**：
```json
[
  {
    "文号": "国发〔2026〕5号",
    "发布主体": "国务院",
    "年份": 2026,
    "序号": 5,
    "类型": "发",
    "联合发文": false
  }
]
```

**支持的文号模式**：

| 模式 | 示例 | 主体推断 |
|------|------|----------|
| `国发〔YYYY〕N号` | 国发〔2026〕5号 | 国务院 |
| `国办发〔YYYY〕N号` | 国办发〔2026〕12号 | 国务院办公厅 |
| `X部令第N号` | 生态环境部令第32号 | 部门规章 |
| `X厅Y函〔YYYY〕N号` | 工信厅联科函〔2026〕193号 | 联合发文（联）、类型推断（函=工作函件） |
| `X政发〔YYYY〕N号` | 浙政发〔2026〕5号 | 省级政府 |
| `X政台发〔YYYY〕N号` | 浙政台发〔2018〕4号 | 省级政府部门 |

**实现**：纯正则匹配，预计 < 60 行 Python，仅依赖 `re` 标准库。不依赖 NLP 模型，不做语义级别的发布机构推断（仅基于文号前缀的编码规则匹配）。

**使用方式**：Claude 将搜索结果文本写入临时文件 → 调用 `python docnumber_parser.py < temp_input.txt` → 读取结构化 JSON 输出 → 在汇编中直接引用。

### 设计原则

1. **工具不做 Claude 能做的事**：搜索、理解、判断、解读由 Claude 完成；工具仅处理机械重复的解析和持久化
2. **零依赖**：所有脚本仅使用 Python 标准库，不引入第三方包
3. **可回退**：工具不可用时（如 Python 环境异常），Claude 应能降级为纯手工处理，不阻断主流程
4. **不自动触发**：每次使用前由 Claude 判断是否有必要调用，不设为固定步骤
