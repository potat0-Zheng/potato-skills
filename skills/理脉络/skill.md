# 理脉络 — 文本逻辑可视化

## 概述

本 skill 处理一个明确的任务：将用户提供的有逻辑结构的自然语言文本，转换为可在 Obsidian 中直接渲染的 Mermaid.js 图表。定位为轻量级纯绘图工具——不做观点回收、不介入知识库管理。

核心原则：**先解构逻辑，再压缩文本，最后生成图表**。

## 图表类型选型

收到文本后，首先判断使用哪种 Mermaid 图表类型。

### 选型标准

| 条件 | 选择 | 说明 |
|------|------|------|
| 文本有明确的递进/因果/时序/层级推导关系 | `graph TD` / `graph LR` | **默认主力**。推导、因果、步骤、层级递进均走此类型 |
| 文本为单一中心主题 + 纯并列子观点，无任何递进关系 | `mindmap` | 仅在无递进时使用 |
| 用户明确要求 mindmap 格式 | `mindmap` | 覆盖自动判断 |

混合结构（如"递进主干 + 并列分支"）：使用 `graph`，以递进关系为主干，并列特性作为分支节点从对应节点发散。

### graph 方向选择

- `graph TD`（上→下）：默认方向，适合大多数递进逻辑
- `graph LR`（左→右）：适合横向对比、时间线

---

## 工作流

```
用户输入文本
    ↓
[一] 逻辑解构 —— 理解逻辑主干，识别断点和结构
    ↓
[二] 节点压缩 —— 基于解构结果，逐节点压缩文本
    ↓
[三] 图表生成 —— 选型、拆分、生成 Mermaid 代码块
    ↓
输出：代码块 + 每张图的逻辑摘要
```

三步之间不跳步。第一步和第二步为内部分析，不产出用户可见内容；第三步产出最终图表和摘要。

---

## 第一步：逻辑解构

在压缩任何文本之前，先对原文做轻量分析。此步骤不产出用户可见内容，仅作为后续压缩和生成的内部依据。此步骤的设计借鉴了 /纳灵光 分析框架中"提取核心洞见 + 识别逻辑断点"的思路。

### 解构三问

**1. 逻辑主干是什么？**
- 识别主逻辑类型：单一递进链 / 递进+并列混合 / 对比结构 / 层级分类
- 如果原文有多段，每段的主逻辑是否不同？

**2. 逻辑断点在哪里？**
- 原文从推导切换到枚举的位置
- 从因果切换到举例的位置
- 从论证切换到总结的位置
- 这些断点即后续拆图的自然切割线

**3. 混合关系是什么？**
- 并列分支是递进节点的展开说明？还是独立的平行论点？
- 展开说明 → 作为分支节点附属于递进节点
- 独立论点 → 考虑拆为独立图表

### 逻辑关系标注

在内部梳理时，显式标注每条边的逻辑类型，供后续步骤选择箭头样式：

| 关系 | 含义 | Mermaid 箭头 | 示例 |
|------|------|-------------|------|
| 推导/递进 | A 推导出 B，A 之后是 B | `-->` | 前提 → 结论 |
| 因果 | A 直接导致 B | `==>` | 原因 ⇒ 结果 |
| 隐含/暗示 | A 暗示 B，非严格推导 | `-.->` | 现象 -.-→ 暗示 |
| 展开/枚举 | A 包含/展开为 B1, B2... | `-->` | 概念 → 特性1,2,3 |
| 对比 | A 与 B 对立/对照 | `-->`（加标签） | 立场A \|对比\| 立场B |

## 第二步：节点压缩

基于第一步的解构结果，逐节点压缩文本。

### 压缩原则

1. **保留命题完整性**：单个节点读出来是一句能成立的话，不是碎词
   - ✓ `孤立商品：无外部意义`
   - ✗ `孤立商品`（太碎，丢失判断）
2. **删修饰从句，留主谓宾骨架**：砍掉定语、状语、举例、修辞
3. **越深越短**：主干节点可保留完整短句（≤20 字），第三层起节点尽量 ≤12 字，末梢 ≤8 字
4. **必要时断行**：单节点文本超过 15 字且无法精简时，用 `<br/>` 断行

### 软性约束

以下为指引，非硬性规定。逻辑需要时允许超出：

- 单图最大深度：尽量 ≤ 4 层
- 同层节点数：尽量 ≤ 6 个
- 单节点文本：尽量 ≤ 15 字

超出时优先考虑拆分，而非无限叠加。

### 压缩示例

**原文**：
> 零散的、孤立的商品几乎无法产生商品本身之外的意义。成组的、成套的商品构成一张难以逃脱的意义网——当你看到微波炉时，你被现代家居的全部意义所俘获。这种意义网对消费者施加一种合法性暗示：对商品组合的思考被认为是自然而然、无需质疑的。

**第一步解构**：对比结构（孤立 vs 成套），成套商品展开为三个子论点（难以逃脱、俘获性、合法性暗示）。

**第二步压缩**：

| 节点 | 文本 | 操作 |
|------|------|------|
| A | `孤立商品` | 留主语 |
| B | `无法产生外部意义` | 砍修饰 |
| C | `成套商品` | 留主语 |
| D | `构成意义网` | 砍修饰 |
| E | `难以逃脱` | 末梢，极短 |
| F | `俘获性：<br/>单品触发整体意义` | `<br/>` 断行 |
| G | `合法性暗示` | 留主谓 |
| H | `消费思考被自然化` | 砍从句 |

## 第三步：图表生成

### 3.1 选型

按图表类型选型标准判定 `graph TD` / `graph LR` / `mindmap`。

### 3.2 拆图

当原文逻辑链路超出单图可呈现范围时，**按逻辑断点拆分**——在第一步识别的断点处切割，每张图聚焦一个逻辑阶段。

拆分规则：
- 在逻辑断点处切割，不在论证中间切断
- 每张图有自己明确的中心主题
- 图与图之间用简短文字承上启下
- 拆分后每张图应独立可读
- 不设最小图数，文本简单就一张图，不为了拆分而拆分

### 3.3 生成 Mermaid 代码块

生成 Obsidian 可直接渲染的 ````mermaid` 代码块。

#### graph 格式规范

- 节点 ID 使用大写字母，保持可读性
- 箭头类型严格对应第一步标注的逻辑关系
- 同一逻辑层级的节点在代码中相邻排列
- 避免交叉连线——节点排列顺序应使连线方向清晰

箭头对照：
- `-->` 推导/递进/展开
- `==>` 因果
- `-.->` 隐含/暗示
- `-->|标签文本|` 需要标注关系时加标签

#### mindmap 格式规范

- 仅在没有递进关系或用户明确要求时使用
- 缩进表示层级
- 中心节点用 `root((中心主题))` 格式

---

## 输出格式

每张图产出以下结构：

```markdown
### 图 N：[中心主题]

*[逻辑摘要：一句话说明这张图覆盖了原文的哪部分逻辑]*

```mermaid
graph TD
    ...
```
```

若有多张图，图与图之间加衔接说明：

```markdown
*[上一张图论证了 X，下图继续展开 X 的后果 Y 和 Z]*
```

## 边界情况处理

| 情况 | 处理 |
|------|------|
| 原文无明显逻辑结构（纯叙述/描写/故事） | 告知用户"未检测到清晰的逻辑结构"，说明原因，询问是否用简单层级分类图呈现 |
| 原文过短（仅一两句话） | 仍尝试生成，单图即可，不强行拆分 |
| 原文过长（完整文章） | 按章节/段落自动拆分，每节独立处理 |
| 原文包含大量例证/数据 | 例证不入图，仅提炼其支撑的论点入图 |
| 原文观点已在知识库某页面存在 | 不检测，不介入——本 skill 只做绘图，不管理知识库 |

---

## 注意事项

- **轻量定位**：本 skill 只做文本到图表的转换。不归档、不索引、不关联 vault。若用户希望将图表中的洞察纳入知识库，建议使用 /纳灵光
- **先解构后压缩**：不得跳过第一步——未经逻辑分析的压缩会丢失结构信息，导致图表徒有形式没有逻辑
- **命题完整性优先于字数**：宁可节点稍长，也不碎成不成句的词组
- **拆分以逻辑为本**：只在逻辑断点处切割，每张子图应独立可读
- **输出格式 Obsidian 原生兼容**：代码块可直接嵌入 Obsidian 页面渲染，无需额外插件
